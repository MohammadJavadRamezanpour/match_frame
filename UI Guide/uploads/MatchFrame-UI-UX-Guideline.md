# MatchFrame — گایدلاین UI/UX (برای کلاد دیزاین)

> این سند مرجع طراحی و ساخت رابط کاربری MatchFrame است. هدف، یک تجربهٔ **آرام، قابل‌اعتماد، مدرن و محترمانه** است.
> کاربر چهرهٔ شخصی خودش را آپلود می‌کند و کمی نگران قضاوت‌شدن است؛ پس هر تصمیم طراحی باید **حس امنیت و جدی‌بودن** بدهد، نه هیجان یا سبکی.

## ۱. فلسفه و شخصیت طراحی

**کانسپت: «اتاق تاریک عکاسی + نقطهٔ تمرکز» (Darkroom & Spotlight)**

عکاسان فضای کار خود را خنثی و خاکستری نگه می‌دارند تا قضاوت رنگ دقیق باشد. ما هم همین کار را می‌کنیم: **محیط، خاکستری خنثی و آرام است** و رنگ گرمِ طلایی فقط برای یک‌چیز کنار گذاشته می‌شود: **برجسته‌کردن بهترین عکس**. این یعنی توجه کاربر دقیقاً به همان‌جا می‌رود که مهم است.

این محصول دربارهٔ همسریابی است، اما **نباید شبیه اپ‌های قرمز/صورتی همسریابی باشد**. آن‌ها هیجان‌محورند؛ ما اعتمادمحور. شخصیت ما: یک مشاور حرفه‌ای، آرام، صادق و مهربان.

* **آرام، نه هیجان‌زده.** فضای سفید زیاد، رنگ‌های خاموش.
* **دقیق، نه شلوغ.** هر چیز سر جای خودش، اعداد و رتبه‌ها خوانا.
* **گرم، نه سرد.** یک سرورق ظریف و طلایی گرما می‌دهد تا حس «انسانی» منتقل شود.

---

## ۲. اصول کلیدی

1. **اعتماد اول از همه.** هر صفحه باید القا کند که عکس‌های کاربر در امان‌اند. نشانه‌های ظریف امنیت (آیکون قفل، متن «private»، شفافیت دربارهٔ AI) همیشه حاضر باشند.
2. **مینیمال و متمرکز.** هر صفحه یک هدف اصلی دارد. عناصر اضافه حذف می‌شوند (قانون شنل: قبل از تمام‌کردن، یک چیز را بردار).
3. **هرگز قضاوت‌گر نباشد.** حتی عکس ضعیف هم با احترام معرفی می‌شود. رنگ قرمز تند برای «بد بودن» عکس ممنوع است.
4. **Dark Mode از روز اول.** نه به‌عنوان وصله، بلکه به‌عنوان بخشی از سیستم رنگ از همان ابتدا.
5. **محتوا راهنماست.** متن‌ها برای کمک به کاربر نوشته می‌شوند، نه برای فروش یا تزئین.

## ۳. سیستم رنگ (Design Tokens)

رنگ‌ها به‌صورت متغیر CSS در `globals.css` تعریف می‌شوند و Tailwind از آن‌ها استفاده می‌کند. **هرگز رنگ ثابت (hardcoded) در کامپوننت‌ها ننویسید.**

سه نقش رنگی داریم:
- **خنثی (Neutral):** محیط، پس‌زمینه، کارت، متن — بدنهٔ اصلی رابط.
- **برند (Iris / Indigo بنفش‌گون):** دکمه‌های اصلی، لینک‌ها، فوکوس — هویت بصری.
- **اسپات‌لایت (طلایی/کهربایی):** **فقط** برای برجسته‌کردن بهترین عکس و برچسب برتر. اگر همه‌جا استفاده شود، معنایش را از دست می‌دهد.

### Light Mode

```css
:root {
  /* محیط و سطوح */
  --bg:            #F6F6F4;   /* خاکستری روشن خنثی (نه کرم) */
  --surface:       #FFFFFF;   /* کارت و پنل */
  --surface-2:     #EFEFEC;   /* سطح فرورفته/خاموش */
  --border:        #E2E2DE;
  --border-strong: #CFCFC9;

  /* متن */
  --ink:           #17181B;   /* متن اصلی */
  --ink-muted:     #5E6066;   /* متن ثانویه */
  --ink-subtle:    #8A8C92;   /* کپشن، placeholder */

  /* برند (Iris) */
  --primary:       #4A40A8;
  --primary-hover: #3D3690;
  --primary-fg:    #FFFFFF;
  --primary-soft:  #ECEAF8;   /* پس‌زمینهٔ ملایم برند */

  /* اسپات‌لایت (طلایی) — فقط برای بهترین عکس */
  --spotlight:     #E0A23C;
  --spotlight-soft:#FBEFD6;
  --spotlight-fg:  #4A3411;

  /* وضعیت‌ها */
  --success:       #3F8F6B;
  --danger:        #C45B52;   /* قرمز ملایم — هرگز تند نباشد */
  --danger-soft:   #F7E4E2;
}
```

### قانون استفاده از رنگ اسپات‌لایت

طلایی **فقط** در این موارد:
- برچسب و قاب «Best Main Profile Photo» (رتبهٔ ۱).
- نشانگر امتیاز/درصد عکس برتر.
- یک accent ظریف در Hero لندینگ.

طلایی **هرگز** برای دکمهٔ معمولی، لینک، یا متن طولانی استفاده نمی‌شود. (به‌خاطر کنتراست، طلایی برای متن کوچک هم مناسب نیست — فقط برای پر‌کردن، حاشیه و آیکون.)

## ۴. Dark Mode

با `next-themes` پیاده می‌شود (کلاس `.dark` روی `<html>`). همان توکن‌ها با مقادیر تیره بازتعریف می‌شوند:

```css
.dark {
  --bg:            #121316;   /* زغالی عمیق، نه مشکی خالص */
  --surface:       #1A1C20;
  --surface-2:     #22242A;
  --border:        #2C2F36;
  --border-strong: #3A3E47;

  --ink:           #ECECEE;
  --ink-muted:     #A6A8AF;
  --ink-subtle:    #7A7D85;

  --primary:       #7A6FE8;   /* روشن‌تر برای کنتراست روی تیره */
  --primary-hover: #8B81EE;
  --primary-fg:    #FFFFFF;
  --primary-soft:  #211F33;

  --spotlight:     #E8B05A;
  --spotlight-soft:#332714;
  --spotlight-fg:  #FBEFD6;

  --success:       #5BA886;
  --danger:        #D6766C;
  --danger-soft:   #2E1E1C;
}
```

**نکات Dark Mode:**
- در حالت تیره، **به‌جای سایهٔ سنگین از مرز (border) و اختلاف جزئی سطح استفاده کنید**؛ سایه‌های بزرگ روی تیره بد دیده می‌شوند.
- عکس‌های کاربر در هر دو حالت روی سطح روشن (یک قاب با پس‌زمینهٔ `--surface`) نمایش داده شوند تا رنگشان دقیق دیده شود.
- از پرش رنگ هنگام بارگذاری جلوگیری کنید (تم را قبل از hydration ست کنید).

## ۵. تایپوگرافی

یک جفت تایپ **عمدی و متمایز** (نه فونت پیش‌فرض همه‌جا). سه نقش:

| نقش | فونت | کاربرد |
|---|---|---|
| **Display** | `Fraunces` (سریف نرم و گرم، با وزن ۴۰۰–۵۰۰) | فقط با احتیاط: تیتر Hero، عنوان گزارش، نام سه عکس برتر. حس «انسانی و مشاوره‌ای» می‌دهد |
| **Body / UI** | `Hanken Grotesk` (سَنس مدرن و خوانا) | تمام بدنهٔ رابط، دکمه، فرم، متن |
| **Data / Mono** | `Geist Mono` (یا `JetBrains Mono`) | اعداد رأی، درصد، رتبه — حس «ابزار دقیق» می‌دهد |

> چرا سریف برای Display؟ چون گزارش باید مثل توصیهٔ یک مشاور دلسوز خوانده شود، نه گزارش خشک ماشینی. اما سریف فقط در تیترهای بزرگ استفاده می‌شود؛ کل UI با سَنس است تا مدرن و تمیز بماند.

با `next/font` بارگذاری شوند. مقیاس تایپ:

```
Display XL  — 48–64px / Fraunces / line-height 1.05   (Hero)
Display L   — 36px    / Fraunces                       (عنوان گزارش)
H1          — 30px    / Hanken 700
H2          — 24px    / Hanken 600
H3          — 20px    / Hanken 600
Body L      — 18px    / Hanken 400                     (متن گزارش)
Body        — 16px    / Hanken 400                     (پیش‌فرض)
Small       — 14px    / Hanken 400/500
Caption     — 13px    / Hanken 500 / رنگ ink-subtle
Data        — 14–16px / Geist Mono                     (اعداد و رتبه)
```

**قواعد:** طول خط متن گزارش حدود ۶۰–۷۵ کاراکتر؛ از تیترهای تمام‌بزرگ (ALL CAPS) فقط برای eyebrow/برچسب کوتاه استفاده شود؛ وزن‌ها محدود و عمدی باشند.

## ۶. فاصله، شعاع و سایه

**فاصله (Spacing):** مقیاس ۴ پیکسلی استاندارد Tailwind: `4, 8, 12, 16, 24, 32, 48, 64, 96`. سخاوتمند باشید؛ فضای سفید بخشی از حس «آرام و قابل‌اعتماد» است.

**شعاع گوشه (Radius):** نرم و یکدست، نه تیز (برخلاف ظاهر روزنامه‌ای) و نه کاملاً قرص:

```
کارت‌ها / قاب عکس   → 16px  (--radius-lg)
دکمه / input        → 10px  (--radius-md)
چیپ / تگ کوچک       → 8px   (--radius-sm)
Badge وضعیت         → pill (گرد کامل) — تنها استثنا
```

**سایه (Shadow):** ملایم، چندلایه و کم‌کنتراست. در Light از سایهٔ نرم، در Dark از border استفاده کنید:

```
shadow-sm  → کارت در حالت عادی
shadow-md  → کارت هنگام hover یا modal
```
از سایه‌های تیره و بزرگ پرهیز کنید؛ با حس مینیمال و آرام نمی‌خوانند.

## ۷. المان امضای طراحی

**«قاب فوکوس دوربین» (Viewfinder Focus Brackets)**

برای برجسته‌کردن بهترین عکس، از چهار براکت گوشه (مثل نشانگر فوکوس دوربین) دور عکس استفاده می‌شود، به‌همراه یک هالهٔ ملایم طلایی. این المان:

```
   ┌─        ─┐
            ★ Best       ← برچسب طلایی
   [   عکس برتر   ]
   └─        ─┘
```

* فقط روی **عکس رتبهٔ ۱** ظاهر می‌شود (در گزارش و در کارت خلاصه).
* با کانسپت عکاسی/کادربندی هم‌خوان است و در ذهن می‌ماند.
* همین یک المان، «جسارت» طراحی ماست؛ بقیهٔ صفحه آرام و خنثی می‌ماند. همه‌جا تکرارش نکنید.

## ۸. کامپوننت‌ها

پایه از **shadcn/ui** است (قابل ویرایش، داخل پروژه). همه با توکن‌های بالا استایل می‌گیرند.

**دکمه (Button):**
- `primary`: پر، رنگ `--primary`، متن سفید — برای اکشن اصلی هر صفحه (یکی در هر صفحه).
- `secondary`: حاشیه‌دار، پس‌زمینهٔ شفاف — اکشن‌های ثانویه.
- `ghost`: بدون حاشیه — اکشن‌های کم‌اهمیت.
- `destructive`: رنگ `--danger` ملایم — فقط برای حذف.
- حالت `loading` با اسپینر و غیرفعال‌شدن داشته باشد.

**کارت (Card):** پس‌زمینهٔ `--surface`، شعاع ۱۶px، حاشیهٔ ظریف، padding سخاوتمند. واحد اصلی چیدمان است.

**Input / فرم:** برچسب همیشه بالای فیلد، شعاع ۱۰px، حالت فوکوس با حلقهٔ `--primary`، پیام خطا زیر فیلد با رنگ `--danger` (ملایم) و آیکون.

**Badge / برچسب عکس:** چیپ گرد کوچک. برچسب‌های گزارش با رنگ معنادار:
- `Best Main Profile Photo` → طلایی (`--spotlight-soft` + متن `--spotlight-fg`).
- `Strong Supporting Photo` → برند ملایم (`--primary-soft`).
- `Lower Priority` / `Not Recommended as First` → خنثی (`--surface-2`)، **نه قرمز**.

**نوار پیشرفت (Processing):** مرحله‌ای، با چک‌مارک برای مراحل تمام‌شده و اسپینر آرام برای مرحلهٔ جاری.

**Toast / اعلان:** برای پیام موفقیت یا خطا؛ کوتاه، با فعل فعال.

## ۹. حرکت و انیمیشن

حرکت باید **هدفمند و آرام** باشد، نه نمایشی. انیمیشن زیاد، حس «ساختهٔ AI» و بی‌اعتمادی می‌دهد.

* **ورود گزارش:** رتبه‌بندی عکس‌ها با fade/slide ملایم و پلکانی ظاهر شود (stagger کوتاه) تا حس «کشف نتیجه» بدهد.
* **صفحهٔ Processing:** انیمیشن آرام و دلگرم‌کننده (چون کاربر منتظر و کمی مضطرب است). از انیمیشن‌های تند و پرش‌دار پرهیز شود.
* **Hover:** تغییرات ظریف (بالا‌آمدن کارت ۱–۲px، تغییر رنگ ملایم دکمه). مدت‌زمان ۱۵۰–۲۵۰ms.
* **اسپات‌لایت عکس برتر:** هالهٔ طلایی می‌تواند یک‌بار به‌نرمی روشن شود (یک‌بار، نه pulse دائمی).
* **`prefers-reduced-motion` همیشه رعایت شود:** برای کاربرانی که حرکت کمتر می‌خواهند، انیمیشن‌ها حذف یا به fade ساده تبدیل شوند.

## ۱۰. ریسپانسیو

موبایل‌اول. تجربه روی موبایل، تبلت و دسکتاپ باید عالی باشد (کاربر احتمالاً از موبایل عکس آپلود می‌کند).

```
موبایل   < 640px    → یک‌ستونه، دکمه‌های تمام‌عرض، آپلود لمسی راحت
تبلت     640–1024px → دو‌ستونه در داشبورد و رتبه‌بندی
دسکتاپ   > 1024px   → چیدمان کامل، عرض محتوا محدود (max-w ~ 1100px) و وسط‌چین
```

**نکات موبایل:**
- ناحیهٔ آپلود بزرگ و لمسی، با امکان انتخاب چند عکس از گالری.
- جابه‌جایی ترتیب عکس‌ها روی موبایل ساده باشد (drag یا دکمهٔ بالا/پایین).
- اکشن اصلی هر صفحه به‌صورت دکمهٔ چسبیده به پایین (sticky) در دسترس باشد.
- اعداد و رتبه‌ها در گزارش روی موبایل هم خوانا بمانند.

## ۱۱. دسترسی‌پذیری

سطح هدف: **WCAG AA**.

* **کنتراست:** متن روی پس‌زمینه حداقل ۴.۵:۱. **رنگ طلایی برای متن کوچک استفاده نشود** (کنتراست کافی ندارد)؛ فقط برای پر‌کردن، حاشیه و آیکون.
* **رنگ تنها نشانه نباشد:** رتبه‌بندی و برچسب‌ها علاوه بر رنگ، **متن و آیکون** هم داشته باشند (مثلاً «#1» و کلمهٔ Best، نه فقط رنگ طلایی).
* **فوکوس کیبورد:** حلقهٔ فوکوس واضح با رنگ `--primary` روی همهٔ عناصر تعاملی.
* **متن جایگزین:** عکس‌های کاربر `alt` معنادار داشته باشند (مثلاً «عکس آپلودی شمارهٔ ۱»).
* **برچسب فرم:** هر input برچسب واقعی داشته باشد (نه فقط placeholder).
* **اندازهٔ هدف لمسی:** حداقل ۴۴×۴۴ پیکسل.
* **`prefers-reduced-motion`** رعایت شود (بخش ۹).

## ۱۲. لحن و زبان متن‌ها

این بخش برای MatchFrame **حیاتی** است، چون موضوع حساس (چهره و جذابیت) است. متن، خودش بخشی از طراحی است.

**اصول لحن:**
- **محترمانه و کمک‌کننده.** هرگز قضاوت‌گر یا تحقیرآمیز نباشید.
- **مثبت و قابل‌اقدام.** حتی نقطه‌ضعف، به‌شکل «چطور بهتر شود» گفته می‌شود.
- **صادق دربارهٔ AI.** شفاف بگویید رأی‌ها شبیه‌سازی AI هستند، نه افراد واقعی.
- **فعل فعال و ساده.** «بهترین عکس را انتخاب کن»، نه جملات منفعل و پیچیده.
- **یک نام برای هر چیز.** اگر دکمه می‌گوید «Start Analysis»، در ادامه هم همان واژه استفاده شود (انسجام).

**نمونه‌های درست در برابر نادرست:**

| ❌ نگو | ✅ بگو |
|---|---|
| «این عکس بد است / جذاب نیست.» | «این عکس به‌عنوان عکس مکمل بهتر عمل می‌کند تا عکس اول، چون چهره کمی کم‌وضوح است.» |
| «نمرهٔ ظاهر شما: ۴/۱۰» | «این عکس در رتبهٔ پایین‌تری قرار گرفت؛ یک عکس با نور طبیعی‌تر می‌تواند نتیجهٔ بهتری بدهد.» |
| «۱۰۰ نفر رأی دادند.» | «۱۰۰ پروفایل شبیه‌سازی‌شده با AI به عکس‌هایت واکنش نشان دادند.» |

## ۱۳. راهنمای صفحه‌به‌صفحه

### Landing Page
- **Hero:** تیتر با فونت `Fraunces`، یک جملهٔ روشن دربارهٔ ارزش محصول، و دکمهٔ اصلی برند. در گوشه یک accent طلایی ظریف.
- ترتیب بخش‌ها: Hero → «چطور کار می‌کند (۳ مرحله)» → نمونهٔ گزارش (با المان اسپات‌لایت روی عکس برتر) → مزایا → **شفاف‌سازی دربارهٔ AI بودن رأی‌دهندگان** (با لحن صادقانه) → قیمت/شروع → FAQ → CTA نهایی.
- نشانه‌های اعتماد (private, secure) دیده شوند.

### Sign Up / Login
- ساده و تک‌ستونه، کارت وسط‌چین. ایمیل/رمز و (در صورت امکان) Google.
- یک جملهٔ کوتاه دربارهٔ امنیت زیر فرم.

### Dashboard
- بالای صفحه: دکمهٔ پررنگ «New Photo Test».
- لیست تحلیل‌های قبلی به‌صورت کارت: تاریخ، تعداد عکس، **Badge وضعیت** (processing/completed/failed) و دکمهٔ «View Report».
- حالت خالی: تصویر/متن دعوت‌کننده برای ساخت اولین تحلیل.

### New Analysis / Upload
- ناحیهٔ آپلود بزرگ و واضح: «۲ تا ۱۰ عکس»، نمایش محدودیت فرمت/حجم.
- Preview عکس‌ها با امکان **حذف و جابه‌جایی ترتیب**.
- انتخاب **مخاطب هدف** رأی‌دهندگان (واضح و ساده).
- دکمهٔ اصلی «Continue to Payment» (فعال‌شدن مشروط به حداقل ۲ عکس).

### Payment Page
- خلاصهٔ سفارش تمیز (تعداد عکس، قیمت)، یک یادآوری کوتاه که رأی‌ها AI هستند، و دکمهٔ اتصال به Stripe.

### Processing Page
- آرام و دلگرم‌کننده. نوار/مراحل با پیام‌ها:
  `Reviewing your photos` → `Simulating audience reactions` → `Ranking your best photos` → `Preparing your report`.
- متن: «این کار معمولاً چند لحظه طول می‌کشد. می‌توانی منتظر بمانی یا بعداً از داشبورد نتیجه را ببینی.»

### Report Page (مهم‌ترین صفحه)
- **Best Photo:** بالای صفحه، با المان امضای **قاب فوکوس + هالهٔ طلایی** و یک جملهٔ روشن دربارهٔ اینکه چرا عکس اول است.
- **Final Ranking:** فهرست همهٔ عکس‌ها با رتبه، تعداد رأی (`Geist Mono`)، درصد، **Badge** و توضیح کوتاه.
- **Top 3:** سه عکس برتر با توضیح کامل‌تر (چرا رأی بیشتری گرفت، چه حسی می‌دهد، جایگاه پیشنهادی).
- **Overall Profile Feedback:** تحلیل کل مجموعه، با لحن محترمانه.
- **Practical Recommendations:** توصیه‌های ساده و مثبت.
- **Email Report CTA:** دکمهٔ ارسال گزارش به ایمیل + امکان بازگشت به داشبورد.
- این صفحه باید **زیبا، واضح و قابل‌اعتماد** باشد؛ بیشترین دقت طراحی اینجاست.

### Account / Settings
- ایمیل، تاریخچهٔ پرداخت‌ها، تنظیمات حریم خصوصی.
- **حذف حساب/داده** به‌صورت اکشن `destructive` با تأیید دومرحله‌ای (Dialog).

### Admin (سبک، برای بعد)
- داشبورد ساده با اعداد کلی: تعداد کاربران، تحلیل‌ها، وضعیت پرداخت‌ها، خطاهای AI، وضعیت Jobها. همان سیستم طراحی، بدون تجمل.

## ۱۴. حالت‌های خاص

برای هر صفحه این سه حالت طراحی شود (اغلب فراموش می‌شوند):

* **Loading:** اسکلتون (skeleton) به‌جای اسپینر تمام‌صفحه؛ حس سرعت بیشتری می‌دهد.
* **Empty:** پیام دعوت‌کننده + یک اکشن مشخص (نه صفحهٔ خالی بی‌روح).
* **Error:** توضیح روشن + قدم بعدی (دکمهٔ Retry یا تماس با پشتیبانی)، با لحن آرام و بدون سرزنش.

برای تحلیل `failed`: پیام محترمانه و مسیر روشن (تلاش دوباره / پشتیبانی).

## ۱۵. چک‌لیست کیفیت طراحی

قبل از تمام‌شدن هر صفحه، این موارد بررسی شود:

- [ ] فقط از توکن‌های رنگ استفاده شده (هیچ رنگ ثابت hardcoded نیست).
- [ ] طلایی فقط برای «بهترین عکس / برچسب برتر» به‌کار رفته، نه جای دیگر.
- [ ] هر صفحه **یک** اکشن اصلی واضح دارد.
- [ ] هر دو حالت Light و Dark تست و درست‌اند.
- [ ] روی موبایل کاملاً قابل‌استفاده است (آپلود، رتبه‌بندی، دکمه‌ها).
- [ ] فوکوس کیبورد روی همهٔ عناصر تعاملی دیده می‌شود.
- [ ] رتبه/برچسب علاوه بر رنگ، متن و آیکون هم دارد.
- [ ] متن‌ها محترمانه، مثبت، با فعل فعال و یک‌دست‌اند.
- [ ] شفافیت دربارهٔ «AI بودن رأی‌دهندگان» در جای مناسب حاضر است.
- [ ] حالت‌های Loading / Empty / Error طراحی شده‌اند.
- [ ] `prefers-reduced-motion` رعایت شده.
- [ ] یک عنصر اضافه حذف شده (قانون شنل).
